# WebSocket Session Affinity Test

A test application demonstrating session affinity for WebSocket services running with multiple replicas behind nginx ingress.

## The Problem

When running a WebSocket service with multiple replicas:
1. A client connects via WebSocket to **Pod A**
2. A third-party agent needs to send an HTTP request that reaches the same **Pod A**
3. Without session affinity, the HTTP request might go to **Pod B**, which has no knowledge of the WebSocket connection

## Solution: Consistent Hash Routing

We use nginx's `upstream-hash-by` annotation to route requests based on a `SESSION_ID` cookie:

```yaml
nginx.ingress.kubernetes.io/upstream-hash-by: "$cookie_SESSION_ID"
```

### How It Works

1. **Client generates SESSION_ID** before connecting (critical!)
2. **Client sends cookie** on WebSocket upgrade: `Cookie: SESSION_ID=<uuid>`
3. **nginx hashes the cookie value** and maps it to a specific pod
4. **All subsequent requests** with the same `SESSION_ID` cookie hash to the same pod

```
Client в”Ђв”ЂWebSocket + Cookieв”Ђв”Ђв–є nginx в”Ђв”Ђhash(SESSION_ID)в”Ђв”Ђв–є Pod A
                                                              в”‚
Agent в”Ђв”ЂHTTP + Same Cookieв”Ђв”Ђв–є nginx в”Ђв”Ђhash(SESSION_ID)в”Ђв”Ђв–є Pod A вњ“
```

### Why Client-Side SESSION_ID Generation Matters

With consistent hashing, nginx hashes whatever value is in the cookie. If the first request has no cookie:
- Empty string hashes to some pod (e.g., Pod B)
- Server assigns SESSION_ID, subsequent requests hash to different pod (e.g., Pod A)
- **Result: WebSocket on Pod B, HTTP on Pod A = broken affinity**

**Solution**: Generate SESSION_ID client-side BEFORE the first request:

```javascript
// Frontend: Generate before WebSocket connect
let sessionId = document.cookie.match(/SESSION_ID=([^;]+)/)?.[1]
if (!sessionId) {
  sessionId = crypto.randomUUID()
  document.cookie = `SESSION_ID=${sessionId}; path=/; max-age=172800`
}
// Browser automatically sends cookie on WebSocket upgrade
const ws = new WebSocket(wsUrl)
```

```python
# Third-party agent: Use known SESSION_ID
requests.post(url, cookies={"SESSION_ID": known_session_id})
```

## Alternative: Session Affinity Cookies

nginx-ingress also supports session affinity via cookies:

```yaml
nginx.ingress.kubernetes.io/affinity: "cookie"
nginx.ingress.kubernetes.io/affinity-mode: "persistent"
nginx.ingress.kubernetes.io/session-cookie-name: "SERVERID"
```

### How Session Affinity Cookies Work

1. Client sends first request (no cookie)
2. nginx routes via round-robin to **Pod A**
3. nginx sets `Set-Cookie: SERVERID=<hash-of-pod-A>`
4. Client includes `SERVERID` cookie on subsequent requests
5. nginx reads cookie and routes to Pod A

### Why We Didn't Use This Approach

| Feature | Session Affinity Cookie | Consistent Hash |
|---------|------------------------|-----------------|
| First request routing | Round-robin | Hash-based |
| Cookie controlled by | nginx | Application |
| Third-party can target pod | No (cookie is opaque hash) | Yes (known SESSION_ID) |
| Use case | Sticky sessions for same client | External agents targeting specific pods |

**Key limitation**: With session affinity cookies, the `SERVERID` cookie is an opaque hash generated by nginx. A third-party agent cannot know which `SERVERID` maps to which pod. They can only use a SESSION_ID they already know.

With consistent hashing on `$cookie_SESSION_ID`:
- The application controls the session ID
- The client can share their SESSION_ID with external agents
- External agents can target the exact pod holding the WebSocket

## Nginx Ingress Configuration

Required annotations for WebSocket + consistent hash routing:

```yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  annotations:
    # WebSocket timeouts (default 60s is too short)
    nginx.ingress.kubernetes.io/proxy-read-timeout: "3600"
    nginx.ingress.kubernetes.io/proxy-send-timeout: "3600"

    # Consistent hash routing on SESSION_ID cookie
    nginx.ingress.kubernetes.io/upstream-hash-by: "$cookie_SESSION_ID"
spec:
  ingressClassName: nginx
  rules:
  - host: your-domain.com
    http:
      paths:
      - backend:
          service:
            name: your-service
            port:
              number: 8080
        pathType: ImplementationSpecific
```

## How nginx Implements Consistent Hashing

When `upstream-hash-by` is configured, nginx-ingress generates a config like:

```nginx
upstream upstream_balancer {
    server 10.0.1.10:8080;  # Pod A
    server 10.0.1.11:8080;  # Pod B

    hash $cookie_SESSION_ID consistent;
}
```

The `consistent` keyword uses ketama consistent hashing:
- Each pod gets multiple virtual nodes on a hash ring
- Cookie value is hashed to a point on the ring
- Request goes to the next pod clockwise from that point
- Adding/removing pods only remaps ~1/n of sessions

## Summary

For third-party agents to target a specific pod holding a WebSocket connection:

1. Use `upstream-hash-by: "$cookie_SESSION_ID"` (not session affinity cookies)
2. Generate SESSION_ID client-side before connecting
3. Share SESSION_ID with agents who need to reach that pod
4. All requests with same SESSION_ID route to same pod
